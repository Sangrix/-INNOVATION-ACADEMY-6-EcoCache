# Retrieval Tuning Result

## 결론

현재 평가셋 기준 권장 설정은 아래와 같습니다.

```text
top_k = 5
threshold = 0.75
chunk_size = 2000
chunk_overlap = 200
```

이 설정은 Top-3 정확도가 가장 높고, 캐시 히트율도 확보하면서 오답 캐시가 발생하지 않았습니다. chunk 설정까지 비교했을 때도 정확도는 유지하면서 평균 지연 시간이 가장 낮게 나왔습니다.

## top_k 비교

`top_k`는 검색 후보를 몇 개까지 가져올지 정하는 값입니다.

- `top_k=3`: 상위 3개 검색 결과만 사용합니다.
- `top_k=5`: 상위 5개 검색 결과까지 사용합니다.

이번 결과에서는 `top_k=5`가 `top_k=3`보다 더 적절했습니다.

| 비교 기준 | top_k=3 | top_k=5 | 판단 |
| --- | --- | --- | --- |
| 최고 Top-1 accuracy | 85.71% | 85.71% | 동일 |
| 최고 Top-3 accuracy | 88.10% | 90.48% | top_k=5 우세 |
| source URL OK | 100% | 100% | 동일 |
| 오답 캐시 없는 최고 캐시율 | 26% | 26% | 동일 |

Top-1 정확도는 같지만, Top-3 정확도에서 `top_k=5`가 더 높았습니다. 웹 데모에서는 답변 생성 시 참고 문서 후보가 충분히 들어오는 것이 중요하므로 `top_k=5`를 우선값으로 둡니다.

## threshold 비교

`threshold`는 QA semantic cache를 사용할지 판단하는 유사도 기준입니다.

- threshold가 낮으면 캐시 히트율은 올라가지만 오답 캐시 위험이 커집니다.
- threshold가 높으면 오답 캐시는 줄지만 캐시 히트율이 떨어집니다.

이번 결과에서는 `threshold=0.75`가 가장 균형적이었습니다.

| threshold | cache hit rate | wrong cache hit | 판단 |
| --- | --- | --- | --- |
| 0.70 | 34% | 1 | 캐시는 잘 맞지만 오답 캐시 발생 |
| 0.75 | 26% | 0 | 정확도와 캐시율 균형이 가장 좋음 |
| 0.80 | 14% | 0 | 안전하지만 캐시율이 낮음 |
| 0.85 | 8% | 0 | 너무 보수적임 |

`threshold=0.70`은 캐시 히트율이 가장 높지만 오답 캐시가 발생했습니다. 실제 웹 데모에서는 틀린 캐시 답변이 사용자에게 바로 보일 수 있으므로 제외합니다.

`threshold=0.80`과 `0.85`는 오답 캐시가 없지만 `0.75` 대비 정확도 이득이 없고 캐시 히트율만 낮아집니다.

## chunk_size / chunk_overlap 비교

`top_k=5`, `threshold=0.75`를 고정하고 chunk 설정만 비교했습니다.

이번 실험에서는 `chunk_size`가 커질수록 `chunk_overlap`도 같이 키웠습니다.

```text
A: chunk_size=1000, chunk_overlap=100
B: chunk_size=1500, chunk_overlap=150
C: chunk_size=2000, chunk_overlap=200
```

`chunk_size`만 키우고 `chunk_overlap`을 100으로 고정하지 않은 이유는, chunk가 길어질수록 경계에서 잘리는 문맥도 함께 길어질 수 있기 때문입니다. 따라서 overlap도 대략 10% 수준으로 같이 증가시키는 방식이 더 자연스럽습니다.

| preset | chunk_size | chunk_overlap | Top-1 accuracy | Top-3 accuracy | cache hit rate | wrong cache hit | source URL OK | avg latency |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| A | 1000 | 100 | 83.33% | 88.10% | 26% | 0 | 100% | 60.6ms |
| B | 1500 | 150 | 85.71% | 90.48% | 26% | 0 | 100% | 116.8ms |
| C | 2000 | 200 | 85.71% | 90.48% | 26% | 0 | 100% | 59.1ms |

결과적으로 A는 지연 시간은 낮지만 Top-1/Top-3 정확도가 떨어졌습니다. B와 C는 정확도가 같았고, C가 평균 지연 시간이 더 낮았습니다.

따라서 chunk 설정은 아래 값을 우선 적용합니다.

```text
chunk_size = 2000
chunk_overlap = 200
```

## 다음 최적화 후보

이번 실험은 `top_k`, `threshold`, `chunk_size`, `chunk_overlap` 중심으로 진행했습니다. 다음 단계에서 추가로 볼 수 있는 값은 아래와 같습니다.

| 항목 | 설명 | 우선순위 |
| --- | --- | --- |
| QA 검색 top_k / 문서 검색 top_k 분리 | QA 캐시는 top1 판단이 핵심이고, 문서 검색은 LLM에 넘길 후보가 중요하므로 값을 분리해볼 수 있습니다. | 높음 |
| metadata filter | 공지/홍보/학사정보 등 검색 범위를 먼저 좁히는 방식입니다. 특정 카테고리 질문에서 실패를 줄일 수 있습니다. | 중간 |
| 실패 케이스 재점검 | 반복 실패한 질문의 정답 문서 기준과 실제 검색 결과를 비교해 데이터 문제인지 검색 문제인지 분리합니다. | 중간 |
| reranker | 1차 검색 결과를 다시 정렬하는 방식입니다. 정확도는 좋아질 수 있지만 구조가 무거워집니다. | 낮음 |
| query rewrite | 사용자의 질문을 검색에 유리하게 다시 쓰는 방식입니다. LLM 연동 이후 검토하는 편이 좋습니다. | 낮음 |

현재 단계에서는 위 최종 적용 후보를 기본값으로 두고, 다음 실험은 `QA 검색 top_k`와 `문서 검색 top_k`를 분리하는 방향이 적절합니다.
